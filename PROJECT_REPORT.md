### Problem

Traditional two-view stereo can only recover a 2.5D depth map from a single viewpoint, leaving the back and occluded regions of an object completely unseen. Naively chaining pairwise stereo across many views accumulates pose drift, producing point clouds that fail to close loops and are geometrically inconsistent. This project addresses the challenge of reconstructing a metrically accurate, colored, full 360° 3D mesh of a real object from a set of calibrated photographs alone — with globally consistent camera poses and sub-pixel reprojection accuracy. The target benchmarks are the Middlebury multi-view datasets (Dino Ring — 48 images, Temple — 312 images, Temple Sparse Ring — 16 images), which are the standard for evaluating multi-view reconstruction pipelines.

### Approach

The pipeline, implemented from scratch in Python, consists of two stages: Structure-from-Motion (SfM) for sparse geometry recovery, followed by Multi-View Stereo (MVS) for dense surface reconstruction. In the SfM stage, SIFT features are detected in every image and matched pairwise using a FLANN KD-tree matcher with Lowe's ratio test. Matches are geometrically verified using Essential-matrix RANSAC (when ground-truth intrinsics from the Middlebury calibration files are available) or Fundamental-matrix RANSAC as a fallback. The best-scoring image pair seeds a two-view reconstruction via DLT triangulation, and remaining cameras are registered incrementally through PnP with RANSAC, with new 3D points triangulated and duplicate-filtered after each registration. A two-pass sparse Levenberg-Marquardt bundle adjustment (Huber robust loss, sparse Jacobian) then jointly optimizes all camera poses and 3D point positions, with statistical outlier removal between passes, to minimize global reprojection error. In the MVS stage, a plane-sweep stereo algorithm generates per-view depth maps using windowed L1 photometric matching across up to 5 source views selected by baseline angle; depth maps are fused via multi-view consistency filtering (a point must agree in at least 3 views within 1% relative depth) and statistical outlier removal. The final dense cloud is meshed using Ball Pivoting surface reconstruction (Open3D) and exported as vertex-colored PLY files.

### Results

The pipeline was evaluated on all three Middlebury benchmarks with MVS densification enabled. The final bundle-adjusted mean reprojection errors are **0.681 px** for Dino Ring (19 cameras, 1.89M dense points, 1.27M mesh triangles), **0.641 px** for Temple (50 cameras, 2.99M dense points, 1.55M mesh triangles), and **0.241 px** for Temple Sparse Ring (4 cameras, 109K dense points, 111K mesh triangles) — all well below the 1-pixel accuracy target set in the project proposal. The primary deliverables are the `.ply` mesh files generated by running the Python pipeline (`test_phase4.py`); these are the quality outputs that capture the full 360° geometry with vertex colors. An interactive Three.js web viewer was also built as an art/presentation component, converting the PLY meshes to Draco-compressed GLB files (~17x smaller) for browser-based exploration — however, the web viewer is purely for visualization and is not the scientific deliverable. To reproduce the quality meshes, the Python program must be run.

### 5. Verify the input datasets

The Middlebury source images should already be present under `Datasets/`:

```
Datasets/dinoRing/           # Dino Ring — 48 images, 640×480, calibration: dinoR_par.txt
Datasets/temple/             # Temple — 312 images, 640×480, calibration: temple_par.txt
Datasets/templeSparseRing/   # Temple Sparse Ring — 16 images, 640×480, calibration: templeSR_par.txt
```

Each dataset folder contains:
- **PNG images** — the source photographs
- **`*_par.txt`** — Middlebury camera calibration (intrinsic K + extrinsic R, t per image). The pipeline auto-detects this via `try_load_intrinsics()` and switches to Essential-matrix RANSAC when it is found.
- **`*_ang.txt`** — viewing angles (used for source-view selection)

### 6. Run the full reconstruction pipeline

The main entry point is `test_phase4.py`, which exposes the following CLI:

```bash
python test_phase4.py [--images DIR] [--max-cameras N] [--mvs]
```

| Flag | Default | Description |
|---|---|---|
| `--images` | `Datasets/dinoRing` | Path to a dataset directory |
| `--max-cameras` | `10` | Maximum cameras to register (the pipeline will register fewer if it runs out of registrable views) |
| `--mvs` | off | Enable MVS densification. **Required to produce the dense meshes reported in Results.** Without this flag, only the sparse point cloud is exported. |

The exact commands used to produce the results reported in this document:

```bash
# Dino Ring (48 images → 19 cameras registered, ~11 min)
python test_phase4.py --images Datasets/dinoRing --max-cameras 50 --mvs

# Temple (312 images → subsampled to 150 → 50 cameras registered, ~57 min)
python test_phase4.py --images Datasets/temple/ --max-cameras 50 --mvs

# Temple Sparse Ring (16 images → 4 cameras registered, ~2 min)
python test_phase4.py --images Datasets/templeSparseRing --max-cameras 50 --mvs
```

> **Note on `--max-cameras 50`:** All three reported runs requested 50 cameras. The pipeline registers as many as it can up to this limit — DinoRing only managed 19 (out of 48 images) before running out of geometrically viable candidates, and TempleSparseRing only had 16 images to begin with (4 registered). Temple's 312 images are uniformly subsampled to 150 (to keep pairwise matching tractable), and 50 of those were registered. The pipeline logs "No more cameras can be registered" when it stops early.

Execution proceeds through the following stages, all logged to the console and to `logs/`:

1. **[1/9] Image loading** — `ImageLoader` reads PNGs, caps the long dimension at 1600 px. If the image count exceeds `3 × max_cameras`, images are uniformly subsampled around the ring (e.g. Temple: 312 → 150 at step 2.1).
2. **[2/9] SIFT feature detection** — `cv2.SIFT_create(nfeatures=2000)`, FLANN KD-tree matching, Lowe's ratio test at 0.75.
3. **[3/9] Pairwise matching + RANSAC** — Essential matrix when `*_par.txt` is found (0.5 px threshold, 99% confidence, 2000 iterations); Fundamental matrix fallback otherwise. Minimum 50 inliers per pair.
4. **[4/9] Two-view seed selection** — Pairs with ≥100 inliers scored by `n_inliers × coverage × inlier_ratio`; top 10 tried in order. `cv2.recoverPose` decomposes E, `cv2.triangulatePoints` produces initial 3D points, filtered by parallax ≥1° and reprojection error ≤2 px. First pair yielding ≥20 valid points becomes the seed.
5. **[5/9] Incremental PnP registration** — Cameras added one-by-one via `cv2.solvePnPRansac` (SOLVEPNP_ITERATIVE, 2.5 px threshold, min 30 inliers). After each registration: new-point triangulation, KD-tree duplicate filtering (0.2% of scene extent), and multi-view observation linking (projects new points into other registered cameras, adds observations if reprojection < 3 px).
6. **[6/9] Bundle adjustment — Pass 1** — `scipy.optimize.least_squares` with Huber loss, sparse Jacobian, `trf` method. Camera closest to origin is fixed. Parameterization: 6-vector (Rodrigues rvec + tvec) per camera + 3D point XYZ. `ftol=1e-8`, `max_nfev=500`.
7. **Statistical outlier filtering** — KD-tree, k=10 neighbors, removes points with mean neighbor distance > mean + 2.5× std.
8. **Bundle adjustment — Pass 2** — Same optimizer, runs on the cleaned point cloud.
9. **[7/9] Sparse point colorization** — For each 3D point, averages the RGB pixel value at its stored observation coordinates across all cameras that saw it.
10. **[8/9] MVS densification** (only with `--mvs`) — Per-view plane-sweep stereo: 128 depth planes, 5×5 box-filtered windowed L1 cost, up to 5 source views selected by 5°–30° baseline angle (ideal ~10°), per-camera depth range from 5th/95th percentile of sparse points. Dark-pixel masking (brightness ≤ 30/255), confidence thresholding (0.6), multi-view consistency (must appear in ≥3 depth maps with < 1% relative depth difference), then statistical outlier removal (k=20, std_ratio=2.0).
11. **[9/9] PLY export** — Points, camera frusta, and metadata (reprojection error, camera count) written as binary PLY.
12. **[10/10] Surface reconstruction** — Ball Pivoting (auto-radii: [1.5, 2.0, 3.0] × mean NN distance) for clouds > 10,000 points; Poisson (octree depth 9) otherwise. Normals estimated via kNN (k=30), oriented on tangent plane. Mesh exported as vertex-colored PLY.

### 7. Where the outputs land

`test_phase4.py` always writes to `data/output/`:

```
data/output/
├── phase4_reconstruction_points.ply    # Dense (or sparse) colored point cloud
├── phase4_reconstruction_cameras.ply   # Camera frusta (one per registered camera)
├── phase4_reconstruction_mesh.ply      # Final colored surface mesh — the deliverable
└── phase4_colored_reconstruction.png   # Matplotlib 3D scatter plot of the reconstruction
```

**Consecutive runs overwrite each other.** To preserve a run, move the outputs before launching the next dataset:

```bash
# After the Dino Ring run:
mkdir -p DinoRing
mv data/output/phase4_reconstruction_*.ply DinoRing/
mv data/output/phase4_colored_reconstruction.png DinoRing/

# After the Temple run:
mkdir -p Temple
mv data/output/phase4_reconstruction_*.ply Temple/
mv data/output/phase4_colored_reconstruction.png Temple/

# After the Temple Sparse Ring run:
mkdir -p TempleSparseRing
mv data/output/phase4_reconstruction_*.ply TempleSparseRing/
mv data/output/phase4_colored_reconstruction.png TempleSparseRing/
```

The top-level `DinoRing/`, `Temple/`, and `TempleSparseRing/` folders in this repo are exactly this — the preserved outputs of the three runs reported in Results. Each contains:
- `phase4_reconstruction_mesh.ply` — the final colored triangle mesh (**the quality deliverable**)
- `phase4_reconstruction_points.ply` — the dense colored point cloud (reprojection error is in the PLY header as a `comment` line)
- `phase4_reconstruction_cameras.ply` — the registered camera frusta
- `phase4_colored_reconstruction.png` — a Matplotlib scatter-plot preview

Open any `.ply` in MeshLab, Blender, CloudCompare, or Open3D to inspect.

### 8. ART component — interactive web viewer

Everything in the `web/` directory is an **ART / presentation layer** on top of the quality PLYs. It is not part of the scientific pipeline; it only exists so the meshes can be explored interactively in a browser. The actual reconstruction accuracy lives in the PLY files produced above.

To regenerate the web assets after producing fresh PLYs:

```bash
# Install Node tooling (one-time)
cd web
npm install
cd ..

# Copy the newly generated quality PLYs into web/models/ under the viewer's names
cp DinoRing/phase4_reconstruction_mesh.ply          web/models/dino.ply
cp Temple/phase4_reconstruction_mesh.ply            web/models/temple.ply
cp TempleSparseRing/phase4_reconstruction_mesh.ply  web/models/templeSparseRing.ply

# Convert PLY → GLB, then Draco-compress in place
python web/tools/convert_ply_to_glb.py
bash   web/tools/compress_glbs.sh
```

The viewer can then be opened locally (`web/index.html`) or deployed as a static site (a `web/vercel.json` is included for one-click Vercel deployment). Draco compression uses 14-bit position quantization, which keeps the geometric error roughly two orders of magnitude below the MVS reconstruction noise floor, so the visual quality is indistinguishable from the source PLY.

Installed via `pip install -r requirements.txt`:

### Disk and memory footprint

- **Datasets**: ~200 MB (Middlebury source images)
- **Quality PLY outputs**: ~100–180 MB per dataset (dense colored mesh)
- **Web-ready GLBs**: ~5–20 MB per dataset (Draco-compressed)
- **Peak RAM during MVS**: ~4–6 GB on Temple at 20 cameras

### Reproducibility notes

- Seed-pair selection is deterministic given a fixed input ordering, so repeated runs on the same dataset produce numerically identical meshes.
- The final two-pass BA reprojection error is written as a `comment` line in the PLY header of `phase4_reconstruction_points.ply` and also logged to `logs/` alongside the per-iteration BA residuals.
- The success criterion is **mean reprojection error < 1 pixel**, matching the project proposal. All three benchmark datasets clear this threshold (0.241 / 0.641 / 0.681 px).
